<objective>
Build `ImageCompressViewModel` — the `@Observable @MainActor` batch orchestrator that drives the results table. It takes the dropped image URLs + quality, runs each image through `ImageCompressTransformer.compress` off the main thread inside an `autoreleasepool` (bounding peak memory under the <100MB target, INFRA-18), updates one `CompressRow` per image live (`.pending`→`.compressing`→`.done`/`.failed`), supports cancellation, conforms to `ToolShortcutActions`, and records one history entry per batch. Also defines the `CompressRow` view-data model the table renders.

Purpose: Mirror `HashViewModel.startFileHash`'s proven off-main Task + progress + cancellation shape, adapted to a multi-image batch loop (D-01) with live per-row updates (D-09). Keeps all ImageIO out of the UI (it stays in the Transformer from plan 01).

Output: `Tools/ImageCompress/ImageCompressViewModel.swift` (including `CompressRow`) and `FlintTests/ImageCompressViewModelTests.swift`, added to their Xcode targets.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dropped URLs → off-main batch loop | Untrusted file set processed concurrently with the UI; one bad file must not abort the batch or the app |
| batch loop → HistoryStore (via onSaveHistory) | Transformation metadata persisted; must contain no secrets and no path-injection surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Denial of Service | batch loop on a malformed file mid-batch | mitigate | Per-image result is a typed `Result`; a `.failure` becomes a `.failed` row and the loop continues. The mixed valid+corrupt unit test proves the batch survives (INFRA-17). |
| T-05-05 | Denial of Service / resource exhaustion | large batch peak memory | mitigate | Each image compressed inside `autoreleasepool` in a sequential off-main loop, bounding peak memory toward the <100MB target (INFRA-18, Pitfall 4). |
| T-05-06 | Information disclosure | history write | mitigate | HistoryEntry stores only filenames + an aggregate savings summary — no secrets exist in this tool (unlike Hash/JWT). No new column; existing schema reused (05-PATTERNS.md line 150). |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | No package installs this phase — zero new external dependencies (D-03). |
</threat_model>
