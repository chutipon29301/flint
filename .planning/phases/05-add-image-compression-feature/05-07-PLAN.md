<objective>
Fix GAP 3 (UAT Test 9) and verify GAP 4 (UAT Test 10) — they are COUPLED. GAP 4 is downstream
runtime residue of GAP 3 (diagnosed in .planning/debug/redrop-produces-no-output.md and

plan fixes cancellation and verifies that a clean re-drop then disambiguates correctly. Do NOT

Root cause (GAP 3): the heavy work runs as synchronous
`autoreleasepool { ImageCompressTransformer.compress(...) }` inside
`await Task.detached(priority:.userInitiated){...}.value` (ImageCompressViewModel.swift lines 173-177).
Two compounding defects:
  1. NON-CANCELLABLE WORK — the transformer has zero Task.isCancelled checks, and Task.detached is
     deliberately disconnected from the parent's cancellation, so cancel() can neither interrupt nor
     even signal the in-flight quantization. A slow PNG runs to completion regardless.
  2. ROW STRANDED — on return, line 181 `guard !Task.isCancelled else { break }` breaks BEFORE
     line 187 `rows[i].apply(result)`, so the row set to .compressing at line 165 never reaches a
     terminal state. The View renders ProgressView for .compressing (ImageCompressView 261-264) → spins forever.
  3. cancel() (lines 235-239) sets isCompressing=false immediately, hiding the Cancel button while work continues.

Fix:
  (a) Add a cooperative Task.isCancelled checkpoint inside the long quantization loop in the
      transformer so in-flight work can actually stop.
  (b) Run that work off-main in a way that PROPAGATES parent cancellation (a child task, NOT
      Task.detached) — so the cooperative check observes cancel().
  (c) On cancel, resolve the .compressing row to a terminal state — do NOT break before apply.
  (d) Keep the Cancel button visible until the in-flight row resolves (don't flip isCompressing eagerly).
  (e) GAP 4: once (a)-(d) land, the orphaned non-cancellable task disappears; verify a clean
      re-drop of the same image disambiguates and writes -compressed-1.

CRITICAL Swift 6 constraint (05-02-SUMMARY auto-fix #1): the outer batch loop MUST stay `Task { }`
(MainActor-bound) so the non-Sendable `onSaveHistory`/`capturedOnSave` closure is never sent across
a non-MainActor boundary. The cancellation fix must NOT reintroduce that data-race error. Off-main
ImageIO is still required (INFRA-18) — achieve it without disconnecting from cancellation.

Purpose: A Cancel button that leaves a row spinning forever and silently masks the next drop is a
trust-breaking bug. After this plan, Cancel stops work and resolves the row, and re-drop works.
Output: Updated ViewModel + transformer cancellation checkpoint + a slow-fixture cancellation test.
