## 0. The One Thing That Will Bite The Planner: ToolRegistry is FROZEN but has no stubs for the 5 new tools

`Core/Services/ToolRegistry.swift` is marked **FROZEN — "Wave-2 tool plans do NOT edit this file"**, and its `tools` array hard-codes exactly the **7 Phase-1 tools**. There are **no pre-registered stubs** for Regex/Color/Markdown/NumberBase/Diff (unlike the Phase-1 wave-1 stubs the comment refers to).

**Implication for the plan:** Phase 2 *must* add the five `*Definition.make()` calls to the `tools` array. This is the single edit to a "frozen" file the phase requires. The planner should:
- Treat the array append as the **one allowed mutation** of ToolRegistry, in a dedicated integration task, ordered AFTER the five tool quads exist (so the build never references a missing `make()`).
- Keep the **detection-chain ordering rule** (CF-04 / Phase-1 D-06, first-match-wins): only **Regex** and **hex-color** get a `detectionPredicate`; the other three pass `nil`. Insert their predicates into the chain in a position that does NOT shadow the existing JSON→JWT→Base64→URL→Timestamp→UUID order — hex-color and regex are both narrow/high-precision, so place them where their predicate won't false-positive ahead of JSON/Base64. Recommend: hex-color early (very specific `#RRGGBB` shape), regex predicate must be conservative (only fire on input that's *obviously* a regex literal, e.g. `/.../flags`) or it will hijack everything — **safest is to give Regex NO detection predicate (search-only) unless a `/…/` literal is detected.**
- `ToolCategory` already has the needed cases (no new category): **Regex→.analysis, Color→.conversion, Markdown→.formatting, NumberBase→.conversion, TextDiff→.analysis** (matches CONTEXT code_context).

## 1. The Per-Tool Quad Pattern (verified, copy it exactly)

Every Phase-1 tool is a 4-file quad under `Tools/<Name>/`. The five new tools replicate it 1:1.

| File | Shape (verified from Base64/JSON) |
|------|-----------------------------------|
| `*Transformer.swift` | `enum *Transformer { static func … }` — **pure, zero UI imports**, `Foundation` only. Returns `Result<…, TransformError>` or plain values. Has a nested `enum TransformError: LocalizedError`. Must **never crash** on bad input (INFRA-17). |
| `*ViewModel.swift` | `@Observable @MainActor final class *ViewModel: ToolShortcutActions`. `var input: String { didSet { scheduleTransform() } }`, `var output`, `var outputDimmed`, `var errorMessage`. Owns a `Debounce` actor instance. History via injected `onSaveHistory: (HistoryEntry) -> Void`. Implements `primaryOutput() -> String?` + `clearInput()`. **Never imports GRDB.** |
| `*View.swift` | SwiftUI view, per-field `CopyButtonView` / `.copyButton()`, `WarningBannerView` / `InlineErrorView` for errors, adopts `ToolShortcutActions` (⌘⇧C / ⌘⌫). |
| `*Definition.swift` | `enum *Definition { static func make() -> ToolDefinition { … } }` — id/name/category/keywords/sfSymbol/detectionPredicate/makeView. |

**Debounce actor** (defined once in `Tools/JSONFormatter/JSONFormatterViewModel.swift`, reuse it — do NOT redefine):
```
actor Debounce: Sendable {
    func schedule(delay: Duration, action: @Sendable @escaping () async -> Void)
    // cancels in-flight task on each call
}
```
Usage in VM: `Task { await debounce.schedule(delay: .milliseconds(150)) { [weak self] in await self?.runTransform() } }`. ponytail: the Debounce actor already exists project-wide — new tools import it, they don't reimplement.

**Test pattern (MANDATORY per tool):** `FlintTests/<Name>TransformerTests.swift`, `final class …: XCTestCase`, `@testable import Flint`, one test per requirement-ID behavior + a round-trip + a no-crash/garbage-input test. The transformer is pure, so tests hit it directly with no UI. **Every `*ViewModel` debounce/last-good logic is UI-state and is NOT unit-tested** — only the pure transformer is. This matches all 9 existing `*TransformerTests`.

**FROZEN, reuse — do NOT modify:**
`Core/Models/ToolDefinition.swift`, `ToolCategory.swift`, `Core/Services/ToolRegistry.swift` (except the §0 array append), `UI/Components/{SyntaxEditorView,CodeDisplayView,WarningBannerView,InlineErrorView,CopyButtonView,ToolShortcutActions}.swift`, `Core/Extensions/View+CopyButton.swift`, `UI/MainWindowView.swift`.

## 3. Regex Tester (RGX-01..04) — the hardest UX constraint

**Files:** `Tools/Regex/{RegexTransformer,RegexViewModel,RegexView,RegexDefinition}.swift` + `FlintTests/RegexTransformerTests.swift`.

**Engine choice:** Use **`NSRegularExpression`** (named + numbered capture groups, flags g/i/m/s/x map to `NSRegularExpression.Options`). SUMMARY notes Swift 5.7 `Regex` avoids catastrophic backtracking but lacks the easy "list all matches + named groups + ranges in the input string" ergonomics and `x`-flag parity; NSRegularExpression is the pragmatic match for RGX-03's results table. The timeout (below) is what guarantees RGX-02's "never freeze," independent of engine.

**Never-freeze design (D-02, Pitfall #8) — non-negotiable:**
- Run evaluation in `Task.detached(priority:.userInitiated)`, **300ms debounce** before launching, **2s timeout** (race the eval Task against `Task.sleep(2s)`; cancel the loser).
- On timeout → set inline warning "Pattern too slow — possible catastrophic backtracking" via `WarningBannerView` (`.warning`), and **keep last-good highlight dimmed** (CF-02 / `outputDimmed`).
- Cancel the in-flight eval on every new keystroke (the existing `Debounce` actor cancels; the eval Task also needs its own cancellation check inside the match loop for very large inputs).
- The transformer (`RegexTransformer.matches(pattern:flags:in:) -> Result<[RegexMatch], TransformError>`) is pure and synchronous; the **timeout/threading lives in the ViewModel**, not the transformer (keeps the transformer unit-testable). `RegexMatch` carries: full-match range, per-group ranges (numbered + named via `range(withName:)`), match index/position.

**Flags (RGX-01):** g (matchAll vs first), i `.caseInsensitive`, m `.anchorsMatchLines`, s `.dotMatchesLineSeparators`, x `.allowCommentsAndWhitespace`.

**Capture-group color coding (D-03, RGX-02):** assign a stable color per group index, highlight in the test-string editor. The editor is the existing `SyntaxEditorView` (NSViewRepresentable) — to overlay match highlights you either (a) layer an attributed-string highlight pass on the NSTextView's textStorage, or (b) render an overlay. Pitfall #5 applies: any programmatic textStorage mutation must guard re-entrancy. Simplest robust approach: keep `SyntaxEditorView` for input, apply highlight attributes to its `textStorage` in a guarded coordinator method (don't reset `.string`).

**Match results table (RGX-03):** index, position, full match, named + numbered groups. Plain SwiftUI `Table`/`List`, collapsible (D-01).

**Replace mode (RGX-04):** `RegexTransformer.substitute(pattern:flags:in:template:)` using `stringByReplacingMatches`. Live substitution preview pane.

**Pattern library (D-04):** static dictionary of email/URL/phone/date/IP patterns, inserted via a "Patterns ▾" `Menu`. Pure data, no research risk.

**Layout (D-01):** vertical stack in popover — pattern field + flag toggles row → test-string editor → collapsible results table → replace section.

## 4. Color Converter (CLR-01..04)

**Files:** `Tools/Color/{ColorTransformer,ColorViewModel,ColorView,ColorDefinition}.swift` + `FlintTests/ColorTransformerTests.swift`.

**Conversion responsibility split:**
- **HEX / RGB / HSL / HSV + alpha:** pure Swift math in `ColorTransformer` (do NOT route through `NSColor` for these — NSColor colorspace conversions are lossy/surprising and hard to unit-test). Implement the standard RGB↔HSL and RGB↔HSV formulas. These are deterministic and **must be the unit-tested core** (known vectors).
- **OKLCH only:** via **ChromaKit**. This is the only conversion that needs the package. OKLCH↔sRGB is CSS Color Level 4 math; do not hand-roll it.
- All formats are **editable rows** (D-05); editing any row recomputes the others live (CF-01, synchronous pure transform — no debounce needed for these, they're cheap). Canonical internal representation: keep a single source-of-truth color (e.g. linear/extended sRGB RGBA doubles) and derive every row from it.

**Out-of-gamut OKLCH (D-08, Pitfall #9):** after ChromaKit OKLCH→sRGB, check each sRGB component ∈ [0,1]. If outside → **gamut-clip to nearest in-sRGB**, show the clipped HEX/RGB so the swatch stays meaningful, and raise a `WarningBannerView` badge "Out of sRGB gamut — clipped". Test against oklch.com / CSS spec reference vectors. `NSColor` silently clamps — do NOT rely on it to detect out-of-gamut; check the raw components yourself.

**Eyedropper (D-06, CLR-02):** `NSColorSampler().show { (nsColor: NSColor?) in … }` — **zero permissions** (not Screen Recording). Async callback on main actor. System panel: SwiftUI `ColorPicker("", selection: $color)` (wraps NSColorPanel) — no extra wiring.

**Sliders (D-06/CLR-03):** R/G/B + H/S/L `Slider`s bound to the canonical color; two-way sync with the editable rows.

**WCAG contrast (D-07, CLR-04):** collapsible section. Formula = `(L1 + 0.05) / (L2 + 0.05)` where L = relative luminance (sRGB → linearize each channel: `c ≤ 0.03928 ? c/12.92 : ((c+0.055)/1.055)^2.4`, then `0.2126R + 0.7152G + 0.0722B`). Thresholds: **AA normal ≥ 4.5, AA large ≥ 3.0, AAA normal ≥ 7.0, AAA large ≥ 4.5.** Pure function → unit-test it. Current color is one swatch; a second `ColorPicker` supplies the other. One tool, one nav entry.

**Detection predicate (CF-04):** hex-color (`#RGB`/`#RRGGBB`/`#RRGGBBAA`) — narrow, safe to place early in the chain.

## 5. Markdown Previewer (MD-01..04)

**Files:** `Tools/Markdown/{MarkdownTransformer,MarkdownViewModel,MarkdownView,MarkdownDefinition}.swift` + `FlintTests/MarkdownTransformerTests.swift`. Plus a **bundled CSS resource** (GitHub-like stylesheet) added to the app bundle.

**Parse → HTML (MD-01):** swift-markdown parses GFM into a `Document` AST (`Markdown.Document(parsing:)`). Walk it with a custom `MarkupVisitor` to emit **controlled HTML** (Pitfall: "WKWebView XSS via raw HTML — AST to controlled HTML emission only"). GFM features required: tables, task lists, strikethrough, fenced code blocks — all supported by cmark-gfm under swift-markdown. Render is **debounced** (heavier op).

**Display (MD-01/MD-02):** `WKWebView` for preview. **Defer WKWebView instantiation to first appearance** (Pitfall #6 — cold-start budget; do NOT instantiate at startup). `WKWebViewConfiguration` with **JavaScript disabled** unless required (XSS hardening; we never need JS for static preview). Editor highlights Markdown syntax (`SyntaxEditorView`); preview highlights fenced code blocks — reuse HighlightSwift via `CodeDisplayView` styling or emit highlighted `<pre>` in the HTML.

**Self-contained styled HTML (D-11, MD-03):** **CSS inlined** into the `<head>` (a `<style>` block read from the bundled stylesheet) so the output is fully offline and presentable. Export targets:
- **Copied HTML** — string to pasteboard.
- **Saved `.html`** — `NSSavePanel` + write string.
- **Saved `.pdf`** — `WKWebView.createPDF(configuration:) { result in … }` (async, main-actor). Note: CONTEXT D-11 **includes PDF in Phase-2 scope** — this overrides SUMMARY's earlier "PDF deferred to 2.1" note. Plan PDF in this phase.

WKWebView is async/main-actor; the export task must await the navigation-finished delegate before calling `createPDF` or the PDF will be blank.

**Toolbar + footer (D-12, MD-04):** formatting toolbar above editor (bold/italic/link/image/code/table inserts — pure string manipulation on the editor binding/selection); thin footer with **word count + reading-time** (reading time = words / ~200 wpm). Word count is a pure function → unit-test it.

**Layout (D-09):** popover = segmented Editor/Preview toggle (one pane); main window = true side-by-side editor|preview.

## 6. Number Base Converter (NUM-01..03)

**Files:** `Tools/NumberBase/{NumberBaseTransformer,NumberBaseViewModel,NumberBaseView,NumberBaseDefinition}.swift` + `FlintTests/NumberBaseTransformerTests.swift`. **Purely synchronous, no debounce needed** (CF-01: cheap). The richest pure-logic unit-test target of the phase.

**Canonical internal representation (NUM-01/02):** a single `UInt64` bit-pattern + width (8/16/32/64) + signed flag. Parse any of bin/oct/dec/hex into that pattern; render all four bases from it. Editing any field re-parses → updates all others live.

**Two's-complement (NUM-02) — the landmine (SUMMARY: "overflow without indicator"):**
- Store the raw bit pattern in `UInt64`; **mask to the active width** (e.g. width 8 → `& 0xFF`).
- Signed decimal rendering: if the top bit of the active width is set, value = `pattern - 2^width` (so 8-bit `0xFF` → `-1`). Use the masked width, not 64.
- Unsigned: render the masked pattern directly.
- **Overflow handling (NUM-03):** when a parsed value exceeds the active width, either clamp/mask with an overflow **indicator badge** (`WarningBannerView`) rather than crashing or silently truncating. Decide-once: mask + indicate (recommended) vs reject. Empty/garbage input → keep last-good, dim (CF-02). **Never crash** (INFRA-17).
- Test matrix: each width × signed/unsigned × {0, max, min, -1, overflow-by-one, empty, invalid-digit-for-base}.

**Bit-field (D-14, NUM-03):** wrapped rows grouped by nibble/byte with bit-index labels, readable at 64-bit in the 480pt popover. Each bit is a toggle button; flipping a bit XORs the pattern bit and re-renders all fields. Nibble-vs-byte grouping is Claude's Discretion.

## 7. Text Diff (DIFF-01..04)

**Files:** `Tools/TextDiff/{TextDiffTransformer,TextDiffViewModel,TextDiffView,TextDiffDefinition}.swift` + `FlintTests/TextDiffTransformerTests.swift`.

**Line-level (DIFF-02):** native **`CollectionDifference`** — split both inputs into `[String]` by line, `changed.difference(from: original)` → `.insert`/`.remove` ops. Zero-dep, fast. Produce a unified model of added/removed/unchanged lines with line numbers.

**Word-level within changed lines (DIFF-02):** **SwiftDiff** (`Diff.diff(text1, text2)` → `.equal`/`.insert`/`.delete` character/word ops) applied **only** to lines paired as a change (a remove+insert at the same logical position). Google diff-match-patch port — use only the diff algorithm, not match/patch.

**Pure transformer:** `TextDiffTransformer.diff(original:changed:ignoreWhitespace:ignoreCase:) -> DiffResult`. Apply ignore-whitespace/ignore-case (DIFF-04) as **normalization before comparison** while preserving original text for display. Unit-test: identical inputs (no diff), pure insert, pure delete, modified line (word-level), ignore-ws/ignore-case toggles, empty inputs.

**Unified patch export (DIFF-03):** emit standard unified-diff format (`@@ -l,s +l,s @@`, ` `/`-`/`+` line prefixes) to pasteboard. Pure string generation → testable.

**Navigation (DIFF-03):** jump next/prev difference = scroll/select to the next changed hunk; UI-state in the ViewModel.

**Layout (D-15):** two stacked editors (Original/Changed) for input (`SyntaxEditorView`); after diff, popover defaults to **unified**, main window to **side-by-side**; toggle always present.

## 9. Pitfall #5 (NSTextView re-render guard) — applies to THREE new editors

Regex test-string editor, Markdown editor, and Diff input editors all use `SyntaxEditorView`. It already implements the guard (`guard textView.string != text else { return }` + async binding update + selection preservation). **Reuse it as-is.** The new risk is **adding highlight attributes** on top:
- Regex capture-group highlighting and Markdown syntax highlighting mutate `textStorage` attributes. Any such mutation must NOT reset `.string` and must be re-entrancy-guarded (a coordinator flag) or it re-triggers `textDidChange` → infinite loop.
- Keep highlight passes attribute-only; never assign `.string` from a highlight pass.

## 10. Validation Architecture

Phase 2 inherits the Phase-1 validation model: **pure transformers are the unit-tested core; ViewModel/View are not unit-tested.** Per-tool test files mirror the 9 existing `*TransformerTests`.

| Tool | Transformer test focus (requirement-backed) | Reference vectors |
|------|----------------------------------------------|-------------------|
| Regex | matches: numbered+named groups, ranges, each flag, replace template, garbage pattern → error (no crash) | known patterns vs known strings |
| Color | RGB↔HSL↔HSV round-trips, HEX parse/emit + alpha, **WCAG ratio formula**, OKLCH↔sRGB + **out-of-gamut detection** | oklch.com / CSS Color 4 spec vectors; WCAG known pairs |
| Markdown | AST→HTML for tables/task-lists/strikethrough/fenced-code; word-count; reading-time | small GFM fixtures |
| NumberBase | **two's-complement matrix** (each width × signed/unsigned × {0,max,min,-1,overflow,empty,invalid}) | hand-computed |
| TextDiff | line insert/delete/modify, word-level in changed line, ignore-ws/ignore-case, **unified-patch output**, empty inputs | hand-authored before/after |

**Non-negotiable, never-crash (INFRA-17):** every transformer needs an explicit garbage/empty/pathological-input test. The Regex 2s-timeout (D-02) is the one behavior that lives in the **ViewModel** and cannot be covered by a pure transformer test — the plan should note it as a manual/UAT check (paste a catastrophic-backtracking pattern; UI stays responsive, warning shows, last-good highlight stays dimmed).

**Reference test vectors to capture at plan time (SUMMARY research flags):**
- OKLCH correctness needs concrete reference values (oklch.com) — bake 2–3 into ColorTransformerTests.
- Regex timeout strategy (NSRegularExpression + Task.detached + 2s) is the chosen approach; no further spike needed given the timeout guarantees safety regardless of engine.

## 11. Recommended Wave / Build Order (planner input)

1. **Wave 1 (parallelizable, independent):** add the 3 SPM packages + bundle the Markdown CSS resource (project-file task) — must land before any tool importing them. Then the five tool quads are **mutually independent** and can each be their own plan: Regex, Color, Markdown, NumberBase, TextDiff. Each plan = transformer (+ tests) → ViewModel → View → Definition.
2. **Wave 2 (integration, depends on all five):** append the five `make()` calls to `ToolRegistry.tools` (the one frozen-file edit, §0), wire detection predicates for Regex + hex-color into the chain, verify search/launcher/history/⌘-shortcuts for all five.

MVP_MODE is ON for this phase (ROADMAP `**Mode:** mvp`) — the planner should organize each tool as a **vertical slice** (transformer→VM→view→registry) that is independently demoable, rather than a horizontal "all transformers, then all views" split. Each tool quad IS a vertical slice. (No Walking Skeleton — that's Phase-1-only.)
