<objective>
Fix UAT Test 16 gap: first Esc intermittently fails to return to the launcher. Stage-1 Esc is
wired ONLY via SwiftUI .onKeyPress(.escape) on the popover root. When the SyntaxEditorView
NSTextView holds first-responder focus (the normal case after the user types/pastes/clicks into
the input), AppKit delivers Esc to the text view, which consumes it via its built-in
cancel/completion handling and never bubbles it to .onKeyPress — so handleEscape() never runs.
Intermittency correlates exactly with editor focus.

Purpose: Make stage-1 Esc reliable regardless of NSTextView focus, preserving the two-stage Esc
contract (D-03).
Output: An AppKit-layer Esc interceptor in SyntaxEditorView that routes to the existing two-stage
handler, with handleEscape() reachable from that interceptor.
</objective>

<task type="auto">
  <name>Task 1: Intercept Esc in the SyntaxEditorView coordinator and broadcast it</name>
  <files>UI/Components/SyntaxEditorView.swift</files>
  <action>
    Make the focused NSTextView forward Esc instead of silently consuming it (debug session
    primary fix: "override cancelOperation:/doCommandBySelector in the SyntaxEditorView
    coordinator and route to the two-stage handler").

    Approach (NSTextViewDelegate route, preferred — no NSTextView subclass needed):
    Implement `func textView(_ textView: NSTextView, doCommandBy selector: Selector) -> Bool` on
    the existing Coordinator. When selector == #selector(NSResponder.cancelOperation(_:)) (this is
    what Esc maps to in a field editor), post NotificationCenter.default.post(name: .escapePressed,
    object: nil) and return true (true = "handled, do not perform default") so the text view does
    NOT eat Esc for its own cancel/completion. For all other selectors return false so normal
    editing keys (newline, tabs, etc.) behave unchanged.

    Declare a new Notification.Name `.escapePressed` (e.g. "lathe.escapePressed"). Put the
    declaration where it is visible to both SyntaxEditorView and MenuBarPopoverView; simplest is a
    Notification.Name extension in this file OR alongside the others in MenuBarPopoverView.swift —
    pick one location and reference it from both (do not declare twice). Keep AppKit import.

    Do NOT change the existing text-binding behavior (the guard textView.string != text loop-break,
    textDidChange async dispatch, accessibility label/role). Only ADD the doCommandBy override.

    Rationale for this over an app-level NSEvent monitor: the delegate route is scoped to the editor,
    avoids a global monitor that could swallow Esc elsewhere, and keeps the fix in the component that
    causes the starvation.
  </action>
