<objective>
Fix GAP 1 (UAT Test 5): the compressor can hand the user a file LARGER than the original.

Root cause (diagnosed, .planning/debug/png-output-larger-than-input.md):
- The PNG path's never-larger guard (writePNGCompressed) compares the quantized output only
  against a TRUECOLOR RE-ENCODE, never against — and never falling back to — the ORIGINAL
  source file. For an already-optimized low-color PNG (Google 2015 logo, 30 KB), the 256-color
  indexed output (46 KB) beats a fresh truecolor re-encode but loses to the original, yet the
  original is never an eligible output → +55% growth shipped.
- The non-PNG ImageIO path (lines 82-105) has NO size guard at all; size is measured AFTER the
  write (lines 107-115) for reporting only. A re-saved JPEG/HEIC/TIFF that grew is handed back as-is.

Fix: make the ORIGINAL SOURCE FILE a writable candidate on BOTH paths. Write the smallest of
{original, quantized, truecolor} on the PNG path; on the non-PNG path, if the re-encode grew,
replace the output with a copy of the original. Preserve D-02 (same-format-out), D-06 (honest
reporting), and INFRA-17 (never crash on bad input). This is a pure-logic defect in a pure
transformer → TDD (write failing tests first).

Purpose: A compressor that grows a file violates the core product promise. After this plan, the
truth "compressed output is never larger than the original" holds on every code path.
Output: Updated ImageCompressTransformer.swift + new failing-first transformer tests.
</objective>
