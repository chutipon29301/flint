## 3. Regex Tester (RGX-01..04) — the hardest UX constraint

**Files:** `Tools/Regex/{RegexTransformer,RegexViewModel,RegexView,RegexDefinition}.swift` + `FlintTests/RegexTransformerTests.swift`.

**Engine choice:** Use **`NSRegularExpression`** (named + numbered capture groups; flags g/i/m/s/x map to `NSRegularExpression.Options`). SUMMARY notes Swift 5.7 `Regex` avoids catastrophic backtracking but lacks the easy "list all matches + named groups + ranges in the input string" ergonomics and `x`-flag parity, and changes flag semantics; NSRegularExpression is the pragmatic match for RGX-03's results table. The timeout (below) is what guarantees RGX-02's "never freeze," independent of engine. **Stay on NSRegularExpression; rely on the timeout, not the engine, for safety.**

**Flags (RGX-01) → options [VERIFIED: Foundation]:** i `.caseInsensitive`, m `.anchorsMatchLines`, s `.dotMatchesLineSeparators`, x `.allowCommentsAndWhitespace`; **g is NOT an option** — it means "enumerate all matches" (`matches(in:)`/`enumerateMatches`) vs "first only" (`firstMatch(in:)`).

**Never-freeze design (D-02, Pitfall #8) — non-negotiable:**
- Run evaluation off the MainActor (`Task` / `Task.detached(priority:.userInitiated)`), **300ms debounce** before launching, **2s timeout** via `withThrowingTaskGroup` racing the eval task against `Task.sleep(for: .seconds(2))`; whoever finishes first wins, `cancelAll()` the loser. [pattern CITED: swhitty Task+Timeout gist]
- On timeout → set inline warning "Pattern too slow — possible catastrophic backtracking" via `WarningBannerView(.warning)`, and **keep last-good highlight dimmed** (CF-02 / `outputDimmed`).
- Cancel the in-flight eval on every new keystroke. **Landmine:** `NSRegularExpression.enumerateMatches` does NOT natively honor Swift task cancellation — check `Task.isCancelled` inside the enumeration block and set `stop.pointee = true`. The 2s wall-clock timeout is the real safety net; because eval runs off the MainActor the UI thread is never blocked even if eval spins slightly past 2s.
- The transformer (`RegexTransformer.matches(pattern:flags:in:) -> Result<[RegexMatch], TransformError>`) is pure and synchronous; the **timeout/threading lives in the ViewModel**, not the transformer (keeps the transformer unit-testable). `RegexMatch` carries: full-match range, per-group ranges (numbered via `match.range(at:)`, named via `match.range(withName:)`, count via `numberOfCaptureGroups`), match index/position.

**Capture-group color coding (D-03, RGX-02):** assign a stable color per group index, highlight in the test-string editor. The editor is the existing `SyntaxEditorView` (NSViewRepresentable) — to overlay match highlights, apply attribute-only passes to the NSTextView's `textStorage` in a guarded coordinator method (don't reset `.string`). See §9 — Pitfall #5 re-entrancy guard.

**Match results table (RGX-03):** index, position, full match, named + numbered groups. Plain SwiftUI `Table`/`List`, collapsible (D-01).

**Replace mode (RGX-04):** `RegexTransformer.substitute(pattern:flags:in:template:)` using `stringByReplacingMatches(in:options:range:withTemplate:)`. Live substitution preview pane.

**Pattern library (D-04):** static dictionary of email/URL/phone/date/IP patterns, inserted via a "Patterns ▾" `Menu`. Pure data, no research risk.

**Layout (D-01):** vertical stack in popover — pattern field + flag toggles row → test-string editor → collapsible results table → replace section.

## 4. Color Converter (CLR-01..04)

**Files:** `Tools/Color/{ColorTransformer,ColorViewModel,ColorView,ColorDefinition}.swift` + `FlintTests/ColorTransformerTests.swift`.

**Conversion responsibility split:**
- **HEX / RGB / HSL / HSV + alpha:** pure Swift math in `ColorTransformer` (do NOT route through `NSColor` for these — NSColor colorspace conversions are lossy/surprising and hard to unit-test). Implement the standard RGB↔HSL and RGB↔HSV formulas (watch the HSV→HSL `0/0` edge when S=0/V=1 — guard divide-by-zero). These are deterministic and **must be the unit-tested core** (known vectors).
- **OKLCH only:** via **ChromaKit** `Color.oklch(l,c,h,opacity:)` / `NSColor.oklch(...)` for the forward direction; reverse (sRGB→OKLCH) is hand-computed via ChromaKit's public `Core` structs or the documented CSS-L4 inverse matrix (A3). This is the only conversion that needs the package; do not hand-roll the forward chain.
- All formats are **editable rows** (D-05); editing any row recomputes the others live (CF-01, synchronous pure transform — no debounce needed for these, they're cheap). **Canonical internal representation: a single source-of-truth sRGB RGBA in 0...1**; derive every row from it (avoid 5 independent `@State` strings — they drift / feedback-loop).

**Out-of-gamut OKLCH (D-08, Pitfall #9):** ChromaKit's `oklch()` returns `NSColor(displayP3Red:…)` — converting to sRGB **silently clamps**. After conversion, compute the **raw (unclamped) sRGB components**; if any channel ∉ [0,1] → set the clipped HEX/RGB from the clamped values so the swatch stays meaningful AND raise a `WarningBannerView(.warning, "Out of sRGB gamut — clipped")`. ChromaKit exposes NO gamut flag — the transformer must do its own range check. Test against oklch.com / CSS spec reference vectors.

**NSColor read safety (Pitfall):** always `nsColor.usingColorSpace(.sRGB)` (optional — handle nil) before `getRed(_:green:blue:alpha:)` / `getHue(_:saturation:brightness:alpha:)`; on system/P3 colors the raw getters return false or wrong values. [VERIFIED: Apple docs]

**Eyedropper (D-06, CLR-02):** `NSColorSampler().show { (nsColor: NSColor?) in … }` — **zero permissions** (NOT Screen Recording; introduced macOS 10.15). Async callback on the main actor; nil if cancelled. System panel: SwiftUI `ColorPicker("", selection: $color)` (wraps NSColorPanel) — no extra wiring. [VERIFIED: Apple docs + WWDC19 S210]

**Sliders (D-06/CLR-03):** R/G/B + H/S/L `Slider`s bound to the canonical color; two-way sync with the editable rows.

**WCAG contrast (D-07, CLR-04):** collapsible section. Formula [VERIFIED: W3C WCAG]:
- linearize each sRGB channel c (0..1): `c ≤ 0.03928 ? c/12.92 : ((c+0.055)/1.055)^2.4`
- `L = 0.2126·R + 0.7152·G + 0.0722·B` (linearized R/G/B)
- `ratio = (Llighter + 0.05) / (Ldarker + 0.05)`
- Thresholds: **AA normal ≥ 4.5, AA large ≥ 3.0, AAA normal ≥ 7.0, AAA large ≥ 4.5.**
Pure function → unit-test it. Current color is one swatch; a second `ColorPicker` supplies the other. One tool, one nav entry.

**Detection predicate (CF-04):** hex-color (`#RGB`/`#RRGGBB`/`#RRGGBBAA`) — narrow, safe to place early in the chain.

## 6. Number Base Converter (NUM-01..03)

**Files:** `Tools/NumberBase/{NumberBaseTransformer,NumberBaseViewModel,NumberBaseView,NumberBaseDefinition}.swift` + `FlintTests/NumberBaseTransformerTests.swift` + `UI/Components/BitFieldView.swift`. **Purely synchronous, no debounce needed** (CF-01: cheap). The richest pure-logic unit-test target of the phase.

**Canonical internal representation (NUM-01/02):** a single `UInt64` bit-pattern + width (8/16/32/64) + signed flag. Parse any of bin/oct/dec/hex into that pattern; render all four bases from it. Editing any field re-parses → updates all others live. Verified primitives [VERIFIED: Swift stdlib]: `String(value, radix: 2/8/10/16, uppercase:)` to emit; `UInt64(string, radix:)` to parse (nil-safe); `Int8/16/32/64(bitPattern:)` to reinterpret signed at width; `canonical & ((1<<width)-1)` to mask; `canonical ^ (1<<bitIndex)` to toggle a bit; `&+`/`&-` overflow operators where wrap is intended.

**Two's-complement (NUM-02) — the landmine (SUMMARY: "overflow without indicator"):**
- Store the raw bit pattern in `UInt64`; **mask to the active width** (e.g. width 8 → `& 0xFF`).
- Signed decimal rendering: if the top bit of the active width is set, value = `pattern - 2^width` (so 8-bit `0xFF` → `-1`). Use the masked width, not 64 — or sign-extend the masked bits via the width's `Int(bitPattern:)`.
- Unsigned: render the masked pattern directly.
- **Overflow handling (NUM-03):** when a parsed value exceeds the active width, mask + show an overflow **indicator badge** (`WarningBannerView`) rather than crashing or silently truncating (recommended over rejecting). Empty/garbage input / `-` with no number / too-large paste → keep last-good, dim (CF-02). **Never crash** (INFRA-17).
- Test matrix: each width × signed/unsigned × {0, max, min, -1, overflow-by-one, empty, invalid-digit-for-base} (Int8 -128/127, Int64 min/max are the boundary cases).

**Bit-field (D-14, NUM-03):** wrapped rows grouped by nibble/byte with bit-index labels, readable at 64-bit in the 480pt popover. Each bit is a toggle button; flipping a bit XORs the pattern bit and re-renders all fields. Nibble-vs-byte grouping is Claude's Discretion.

## 7. Text Diff (DIFF-01..04)

**Files:** `Tools/TextDiff/{TextDiffTransformer,TextDiffViewModel,TextDiffView,TextDiffDefinition}.swift` + `FlintTests/TextDiffTransformerTests.swift` + vendored `Tools/TextDiff/SwiftDiff/` (see §2).

**Line-level (DIFF-02):** native **`CollectionDifference`** — split both inputs into `[String]` by line, `changed.difference(from: original)` → `.insert`/`.remove` ops. Zero-dep, fast (Myers, SE-0240). Produce a unified model of added/removed/unchanged lines with line numbers.

Unit-test: identical inputs (no diff), pure insert, pure delete, modified line (word-level), ignore-ws/ignore-case toggles, empty inputs.

**Unified patch export (DIFF-03):** emit standard unified-diff format (`@@ -l,s +l,s @@`, ` `/`-`/`+` line prefixes) to pasteboard. Pure string generation → testable.

**Navigation (DIFF-03):** jump next/prev difference = scroll/select to the next changed hunk; UI-state in the ViewModel.

## 9. Pitfall #5 (NSTextView re-render guard) — applies to THREE new editors

Regex test-string editor, Markdown editor, and Diff input editors all use `SyntaxEditorView`. It already implements the guard (`guard textView.string != text else { return }` + async binding update via `DispatchQueue.main.async` + selection preservation). **Reuse it as-is.** [VERIFIED: SyntaxEditorView.swift read] The new risk is **adding highlight attributes** on top:
- Regex capture-group highlighting and Markdown syntax highlighting mutate `textStorage` attributes. Any such mutation must NOT reset `.string` and must be re-entrancy-guarded (a coordinator flag) or it re-triggers `textDidChange` → infinite loop.
- Keep highlight passes attribute-only; never assign `.string` from a highlight pass.
- The new `WebPreviewView` (WKWebView) is the same trap in different clothing — guard identical HTML before `loadHTMLString`.
